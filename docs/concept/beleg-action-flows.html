Beleg-Aktionsflüsse
als ideales Zielbild

Eine vollständige, idealtypische Landkarte aller Aktionen, die System, Teamlead und Mitarbeiter auf einem Beleg (und seinen Positionen, SKU-Zeilen, Boxen, Bündeln, Problemen und ZST-Datensätzen) ausführen — gruppiert nach Lebenszyklus-Phase und verknüpft mit dem Typ (Entität), auf den jede Aktion wirkt.

Dies ist das angestrebte Konzept, nicht der heutige Implementierungsstand: Flüsse sind durchgängig gedacht (Scan-Validierung, serverseitige Arbeitsverifikation, lückenloser Audit-Trail, geschlossene Endzustände). Es zeigt, wie es sein soll — und wie jede Aktion an Entitätstyp und Zustand hängt.

Akteur

System / Ingestion Teamlead Mitarbeiter

Entitätstyp

DocumentSet Beleg Position SKU-Zeile Box Bündel Problem ZST

Lesehilfe

event = Audit-Event (§7.2) Phase-Tag = Lebenszyklus-Gruppe (5) Micro-Chip = betroffener Typ
01

Typen & ihre Beziehung

Auf welche Entität jede Aktion wirkt — die „relation to type".

DocumentSet
Lieferschein + Arbeitsanweisung + WE-Beleg, gruppiert beim Ingest. Quelle der Wahrheit pro Wareneingang.
Beleg (GoodsReceiptCase)
Die zentrale Arbeitseinheit. Trägt Status (§7.1), Priorität, Aufwand, Lagerplatz. 1 DocumentSet → 1 Beleg.
Position → SKU-Zeile
Beleg → n Positionen → n SKU-Zeilen (EAN · Größe · Soll/Ist). Granularität der Mengenkontrolle.
Transport-Box
Eine Box je Shopbereich/Etage (Box-Splitting). Trägt Boxzettel-/Plombe-/Band-Status.
Bündel (AssignmentBundle)
Rollwagen eines Mitarbeiters: n Belege + Abhol-Reihenfolge (RouteStops). Von der Engine erzeugt.
Problem (Issue)
Scope: Beleg / Position / SKU / Box. Blockiert genau seinen Scope bis zur Teamlead-Lösung.
ZST-Datensatz
Bucht fertige Menge + Aufwand (Voll/Teil). Exportiert → Altsystem (zst_done).
Faustregel
Teamlead-Aktionen wirken auf Beleg oder Bündel.
Mitarbeiter-Aktionen wirken granular auf Position, SKU, Box — und schließen den Beleg via ZST ab.
System erzeugt DocumentSet & Beleg.

04

Idealer Fluss je Akteur

Die zwei Hauptpfade durchgängig gedacht — mit Guardrails.

Mitarbeiter — Belegabarbeitung

  • Tagesstart & Paket
    Bündel + verbindliche Abholreihenfolge. Gerät-Problem-Shortcut.
  • Lagerplatz scannen
    picking. Guard: Soll/Ist-Abgleich
  • Vorbereitung & Druck
    Etiketten zuerst. Guard: Druck vor Auspacken
  • Position + SKU prüfen
    Je Position/SKU bestätigt. Guard: Mindest-Stückzahl immer
  • Boxen abschließen
    Zettel → Plombe → Band, je Shopbereich.
  • ZST / Teilabschluss
    completed. Gate: alles bestätigt & verplombt

Teamlead — Steuerung

  • Automatik / Vorschlag
    Engine verteilt; Ausnahmen surfacen. Vorschlag→Übernehmen.
  • Pool pflegen
    Priorisieren · Parken · Stornieren (Grund+Audit).
  • Bündel-Eingriff
    Entziehen/Hinzufügen/Reihenfolge/Pause — optimistisch + Rollback.
  • Probleme triagen
    issue_open lösen → freigeben → zurück in Arbeit. Guard: kein Beleg bleibt stecken
  • Belegdetail prüfen
    Phasen-Scope, Positionen/Boxen, Abschluss-/Problem-Tab.
  • Tagesabschluss
    completed → zst_done, ZST-CSV an Altsystem.
